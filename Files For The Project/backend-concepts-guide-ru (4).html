MediatR

Медиатор-паттерн для развязки компонентов

// Что такое MediatR?

MediatR — это библиотека для .NET, реализующая паттерн Mediator (Посредник). Идея простая: вместо того чтобы один класс напрямую вызывал методы другого, оба общаются через посредника. Это делает код независимым — компоненты ничего не знают друг о друге.

Представьте диспетчерскую в аэропорту. Самолёты не разговаривают друг с другом напрямую — всё идёт через диспетчера. MediatR — это и есть тот диспетчер для ваших классов.

Главная идея Отправитель посылает сообщение (запрос, команду, событие) — и понятия не имеет, кто его обработает. Обработчик получает сообщение — и не знает, кто его отправил. Посредник (MediatR) связывает их.

// Три типа сообщений

IRequest<TResponse>

Запрос с ответом. Один отправитель → один обработчик. Подходит для CQRS-команд и запросов.

INotification

Уведомление без ответа. Один отправитель → много обработчиков. Паттерн «Издатель–Подписчик».

IStreamRequest<T>

Стриминговый запрос. Возвращает IAsyncEnumerable — поток результатов.

IPipelineBehavior

Middleware для запросов. Перехватывает все запросы до/после обработчика (логирование, валидация).

// Pipeline Behavior (Middleware для запросов)

Один из самых мощных инструментов MediatR — возможность перехватить любой запрос до и после его обработки.

// Поведение логирования — оборачивает КАЖДЫЙ запрос
public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> _logger;

    public LoggingBehavior(ILogger<LoggingBehavior<TRequest, TResponse>> logger)
        => _logger = logger;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        var requestName = typeof(TRequest).Name;
        _logger.LogInformation("→ Handling {Request}", requestName);
        
        var sw = Stopwatch.StartNew();
        var response = await next(); // вызов следующего в цепочке
        sw.Stop();
        
        _logger.LogInformation("← Handled {Request} in {Ms}ms",
            requestName, sw.ElapsedMilliseconds);
        return response;
    }
}

Redis

In-memory хранилище данных — кэш, очередь, pub/sub

// Что такое Redis?

Redis (Remote Dictionary Server) — это сверхбыстрое хранилище данных в оперативной памяти. Он работает как база данных типа «ключ–значение», но умеет хранить не только строки, но и списки, множества, хэш-таблицы, sorted sets и многое другое.

Ключевое отличие от обычных БД: Redis хранит данные в RAM, поэтому операции выполняются за микросекунды, а не миллисекунды.

→

Ответ за ~0.1ms

// Основные сценарии использования

  • Кэширование — кэш результатов запросов к БД, HTML-страниц, вычислений
  • Сессии — хранение сессий пользователей (быстрее, чем в БД)
  • Rate Limiting — счётчики запросов с автоматическим истечением (TTL)
  • Pub/Sub — система публикации и подписки на события в реальном времени
  • Очереди — работа с фоновыми задачами через Streams или Lists
  • Distributed Lock — блокировки в распределённых системах

// Кэш-сервис с паттерном Cache-Aside

public class ProductCacheService
{
    private readonly IDistributedCache _cache;
    private readonly IProductRepository _repo;

    public ProductCacheService(IDistributedCache cache, IProductRepository repo)
    {
        _cache = cache;
        _repo = repo;
    }

    public async Task<Product?> GetProductAsync(int id)
    {
        var cacheKey = $"product:{id}";

        // 1. Пробуем взять из кэша
        var cached = await _cache.GetStringAsync(cacheKey);
        if (cached is not null)
            return JsonSerializer.Deserialize<Product>(cached);

        // 2. Если нет в кэше — берём из БД
        var product = await _repo.GetByIdAsync(id);
        if (product is null) return null;

        // 3. Сохраняем в кэш на 10 минут
        await _cache.SetStringAsync(cacheKey,
            JsonSerializer.Serialize(product),
            new DistributedCacheEntryOptions
            {
                AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(10)
            });

        return product;
    }

    public async Task InvalidateCacheAsync(int id)
    {
        // Инвалидируем кэш при обновлении продукта
        await _cache.RemoveAsync($"product:{id}");
    }
}

public class RedisDirectService
{
    private readonly IDatabase _db;

    public RedisDirectService(IConnectionMultiplexer mux)
        => _db = mux.GetDatabase();

    // Строки
    public async Task SetStringAsync()
    {
        await _db.StringSetAsync("mykey", "myvalue", TimeSpan.FromHours(1));
        var val = await _db.StringGetAsync("mykey");
    }

    // Атомарный инкремент (идеально для счётчиков/rate limit)
    public async Task<long> IncrementCounterAsync(string key)
    {
        var count = await _db.StringIncrementAsync(key);
        // Устанавливаем TTL при первом инкременте
        if (count == 1)
            await _db.KeyExpireAsync(key, TimeSpan.FromMinutes(1));
        return count;
    }

    // Хэш (как словарь объекта)
    public async Task SetHashAsync()
    {
        await _db.HashSetAsync("user:42", new[]
        {
            new HashEntry("name", "Alice"),
            new HashEntry("email", "alice@example.com"),
        });
        var name = await _db.HashGetAsync("user:42", "name");
    }

    // Distributed Lock — только один процесс выполняет задачу
    public async Task WithLockAsync()
    {
        var token = Guid.NewGuid().ToString();
        var acquired = await _db.StringSetAsync(
            "lock:my-resource", token,
            TimeSpan.FromSeconds(30),
            When.NotExists); // NX — только если ключа нет

        if (!acquired) return; // кто-то уже держит блокировку
        try
        {
            // критическая секция
        }
        finally
        {
            await _db.KeyDeleteAsync("lock:my-resource");
        }
    }
}

WebHooks

HTTP-коллбэки — "обратный вызов" от сервера к серверу

// Что такое WebHook?

WebHook — это механизм, при котором один сервис автоматически отправляет HTTP POST-запрос на URL другого сервиса в момент, когда происходит какое-то событие.

Аналогия: вы оставляете кому-то свой номер телефона (URL) и просите: «Когда что-то произойдёт — позвони мне». Это и есть webhook — вместо того чтобы постоянно спрашивать «всё готово?» (polling), вы получаете уведомление.

Ваш сервер

Обработка логики

Polling (плохо)

Каждые 5 секунд: «Платёж прошёл?»... «Нет»... «Платёж прошёл?»... Тратит ресурсы зря.

WebHook (хорошо)

Stripe сам уведомит вас одним запросом в момент, когда платёж совершился.

// Пример: принимаем WebHook от Stripe

[ApiController]
[Route("webhooks")]
public class WebHooksController : ControllerBase
{
    private readonly ILogger<WebHooksController> _logger;
    private const string StripeSecret = "whsec_ваш_секрет_из_stripe";

    public WebHooksController(ILogger<WebHooksController> logger)
        => _logger = logger;

    [HttpPost("stripe")]
    public async Task<IActionResult> HandleStripeWebHook()
    {
        // 1. Читаем сырое тело запроса
        var json = await new StreamReader(Request.Body).ReadToEndAsync();

        // 2. Верифицируем подпись — ОБЯЗАТЕЛЬНО
        var signature = Request.Headers["Stripe-Signature"];
        Event stripeEvent;
        try
        {
            stripeEvent = EventUtility.ConstructEvent(json, signature, StripeSecret);
        }
        catch (StripeException e)
        {
            _logger.LogWarning("Неверная подпись Stripe: {Error}", e.Message);
            return BadRequest();
        }

        // 3. Обрабатываем тип события
        switch (stripeEvent.Type)
        {
            case "payment_intent.succeeded":
                var paymentIntent = stripeEvent.Data.Object as PaymentIntent;
                _logger.LogInformation("Оплата {Id} прошла успешно", paymentIntent?.Id);
                // обновить заказ, выслать чек и т.д.
                break;

            case "customer.subscription.deleted":
                var sub = stripeEvent.Data.Object as Subscription;
                // деактивировать подписку пользователя
                break;
        }

        // 4. Возвращаем 200 — иначе Stripe будет повторять запрос
        return Ok();
    }
}

// Собственный WebHook — отправка уведомлений

Если вы создаёте платформу и хотите уведомлять клиентов о событиях — нужно реализовать отправку webhook'ов.

public class WebHookSender
{
    private readonly HttpClient _http;
    private readonly ILogger<WebHookSender> _logger;

    public WebHookSender(IHttpClientFactory factory, ILogger<WebHookSender> logger)
    {
        _http = factory.CreateClient();
        _logger = logger;
    }

    public async Task SendAsync(string targetUrl, object payload, string secret)
    {
        var json = JsonSerializer.Serialize(payload);
        
        // Подписываем payload — получатель может проверить подлинность
        var signature = ComputeHmacSha256(json, secret);

        var request = new HttpRequestMessage(HttpMethod.Post, targetUrl)
        {
            Content = new StringContent(json, Encoding.UTF8, "application/json")
        };
        request.Headers.Add("X-Webhook-Signature", signature);
        request.Headers.Add("X-Webhook-Event", "user.created");

        try
        {
            var response = await _http.SendAsync(request);
            response.EnsureSuccessStatusCode();
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Ошибка отправки webhook на {Url}", targetUrl);
            // Здесь стоит добавить retry-логику (Polly)
        }
    }

    private static string ComputeHmacSha256(string payload, string secret)
    {
        using var hmac = new HMACSHA256(Encoding.UTF8.GetBytes(secret));
        var hash = hmac.ComputeHash(Encoding.UTF8.GetBytes(payload));
        return Convert.ToHexString(hash).ToLower();
    }
}

public class WebHookEventDispatcher
{
    private readonly IWebHookSubscriptionRepository _subscriptions;
    private readonly WebHookSender _sender;
    private readonly ILogger<WebHookEventDispatcher> _logger;

    public WebHookEventDispatcher(
        IWebHookSubscriptionRepository subscriptions,
        WebHookSender sender,
        ILogger<WebHookEventDispatcher> logger)
    {
        _subscriptions = subscriptions;
        _sender = sender;
        _logger = logger;
    }

    public async Task DispatchAsync(WebHookEvent webhookEvent)
    {
        // Ищем все активные подписки на этот тип события
        var subscribers = await _subscriptions
            .GetActiveSubscribersForEventAsync(webhookEvent.EventType);

        var envelope = new
        {
            id = webhookEvent.Id,
            type = webhookEvent.EventType,
            created_at = webhookEvent.CreatedAt,
            data = webhookEvent.Data
        };

        // Отправляем параллельно всем подписчикам
        var tasks = subscribers.Select(sub =>
            _sender.SendAsync(sub.TargetUrl, envelope, sub.Secret));

        await Task.WhenAll(tasks);

        _logger.LogInformation(
            "Dispatched {Event} to {Count} subscribers",
            webhookEvent.EventType, subscribers.Count());
    }
}

    {

    }
}

Rate Limiting

Ограничение частоты запросов — защита от перегрузок и злоупотреблений

// Что такое Rate Limiting?

Rate Limiting — это механизм ограничения количества запросов, которые клиент может отправить за определённое время. Это защищает ваш сервер от DDoS-атак, брутфорса, случайной перегрузки.

Fixed Window

100 запросов в минуту. Окно фиксировано (00:00–01:00, 01:00–02:00). Просто в реализации.

Sliding Window

100 запросов за последние 60 секунд. Плавно считает — нет пиков на стыке окон.

Token Bucket

        opt.AutoReplenishment = true;

    options.OnRejected = async (context, token) =>
    {

        context.HttpContext.Response.Headers["Retry-After"] = "60";

// Rate Limiting по пользователю (через Redis)

Встроенный Rate Limiter — in-memory (на один инстанс). Для распределённых систем нужен Redis.

public class RedisRateLimiterService
{
    private readonly IDatabase _redis;

    public RedisRateLimiterService(IConnectionMultiplexer mux)
        => _redis = mux.GetDatabase();

    /// Возвращает false, если лимит превышен
    public async Task<bool> IsAllowedAsync(
        string userId,
        int maxRequests = 100,
        int windowSeconds = 60)
    {
        var key = $"ratelimit:{userId}:{DateTime.UtcNow:yyyyMMddHHmm}";
        
        // Атомарная операция: INCR + EXPIRE
        var count = await _redis.StringIncrementAsync(key);
        if (count == 1)
            await _redis.KeyExpireAsync(key, TimeSpan.FromSeconds(windowSeconds));

        return count <= maxRequests;
    }
}

// Middleware для проверки
public class RedisRateLimitMiddleware
{
    private readonly RequestDelegate _next;

    public RedisRateLimitMiddleware(RequestDelegate next) => _next = next;

    public async Task InvokeAsync(HttpContext context, RedisRateLimiterService limiter)
    {
        var userId = context.User?.Identity?.Name ?? context.Connection.RemoteIpAddress?.ToString() ?? "anon";

        if (!await limiter.IsAllowedAsync(userId))
        {
            context.Response.StatusCode = 429;
            await context.Response.WriteAsJsonAsync(new { error = "Rate limit exceeded" });
            return;
        }

        await _next(context);
    }
}

// Что такое OAuth 2.0?

OAuth 2.0 — это протокол авторизации (не аутентификации!), который позволяет вашему приложению получить доступ к ресурсам пользователя на другом сервисе, не зная его пароля.

public class JwtTokenService
{
    private readonly IConfiguration _config;

    public JwtTokenService(IConfiguration config) => _config = config;

    public string GenerateAccessToken(User user)
    {
        var key = new SymmetricSecurityKey(
            Encoding.UTF8.GetBytes(_config["Jwt:Secret"]!));

        var claims = new[]
        {
            new Claim(ClaimTypes.NameIdentifier, user.Id.ToString()),
            new Claim(ClaimTypes.Email, user.Email),
            new Claim(ClaimTypes.Name, user.Name),
            new Claim("role", user.Role),
            // Можно добавить любые custom claims
        };

        var token = new JwtSecurityToken(
            issuer: _config["Jwt:Issuer"],
            audience: _config["Jwt:Audience"],
            claims: claims,
            expires: DateTime.UtcNow.AddMinutes(15), // короткое время жизни!
            signingCredentials: new SigningCredentials(
                key, SecurityAlgorithms.HmacSha256));

        return new JwtSecurityTokenHandler().WriteToken(token);
    }

    public string GenerateRefreshToken()
    {
        // Refresh token — случайная строка, хранится в БД
        var bytes = new byte[64];
        RandomNumberGenerator.Fill(bytes);
        return Convert.ToBase64String(bytes);
    }
}

Cross-Origin Resource Sharing — защита браузеров от межсайтовых запросов

// Что такое CORS?

CORS (Cross-Origin Resource Sharing) — это механизм браузера, который блокирует JavaScript-запросы на другой домен, если сервер явно не разрешил это.

Когда ваш фронтенд (https://app.com) пытается запросить API (https://api.com), браузер сначала спрашивает у сервера: «Ты разрешаешь запросы с app.com?». Если сервер не ответит с нужными заголовками — браузер заблокирует запрос.

Важно понимать CORS — это защита БРАУЗЕРА, не сервера. Postman, curl и бэкенд-сервисы CORS не соблюдают. Если запрос не работает только из браузера — это CORS.

Для «сложных» запросов (не GET/POST без кастомных заголовков) браузер сначала шлёт OPTIONS-запрос — «preflight».

// Настройка CORS в Program.cs

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddCors(options =>
{
    // Политика для разработки — разрешаем всё
    options.AddPolicy("Development", policy =>
    {
        policy
            .AllowAnyOrigin()    // любой источник
            .AllowAnyMethod()    // GET, POST, PUT, DELETE...
            .AllowAnyHeader();   // любые заголовки
    });

    // Политика для продакшна — только наш фронтенд
    options.AddPolicy("Production", policy =>
    {
        policy
            .WithOrigins(
                "https://app.yoursite.com",
                "https://yoursite.com"
            )
            .WithMethods("GET", "POST", "PUT", "DELETE", "PATCH")
            .WithHeaders(
                "Authorization",
                "Content-Type",
                "X-Requested-With"
            )
            .AllowCredentials()          // разрешаем cookies/credentials
            .SetPreflightMaxAge(TimeSpan.FromHours(1)) // кэш preflight
            .WithExposedHeaders("X-Total-Count"); // заголовки видные JS
    });

    // Политика для мобильных приложений / постмана
    options.AddPolicy("MobileApp", policy =>
    {
        policy
            .WithOrigins("capacitor://localhost", "ionic://localhost")
            .AllowAnyMethod()
            .AllowAnyHeader();
    });
});

var app = builder.Build();

SignalR

Двусторонняя связь в реальном времени — чаты, уведомления, live-обновления

// Что такое SignalR?

SignalR — это библиотека ASP.NET Core для организации двусторонней связи между сервером и клиентом в реальном времени. Вместо того чтобы клиент постоянно спрашивал сервер «есть ли новые данные?», сервер сам мгновенно отправляет данные всем подключённым клиентам в момент, когда что-то произошло.

SignalR автоматически выбирает лучший транспорт: WebSockets (идеально, постоянное соединение), Server-Sent Events или Long Polling — в зависимости от возможностей браузера и сервера.

Главная идея SignalR создаёт постоянный канал между браузером и сервером. Сервер может в любой момент «толкнуть» (push) данные клиенту — без запроса с его стороны. Клиент тоже может вызывать методы на сервере напрямую.

// Когда нужен SignalR?

💬 Чат и мессенджеры

Сообщения доставляются мгновенно всем участникам комнаты без перезагрузки страницы.

🔔 Live-уведомления

«У вас новый заказ», «Файл загружен», «Кто-то упомянул вас» — в реальном времени.

📊 Live-дашборды

Графики, метрики, статусы заказов обновляются автоматически без F5.

🎮 Коллаборация

Совместное редактирование документов, онлайн-игры, whiteboard в реальном времени.

    public async Task SendMessage(string room, string message)
    {
        var user = Context.User?.Identity?.Name ?? "Anonymous";
        _logger.LogInformation("{User} sent message to {Room}", user, room);

        // Отправляем всем в комнате, кроме отправителя
        await Clients.OthersInGroup(room).SendAsync("ReceiveMessage", new
        {
            user,
            message,
            timestamp = DateTime.UtcNow
        });
    }

    // Присоединиться к комнате
    public async Task JoinRoom(string room)
    {
        await Groups.AddToGroupAsync(Context.ConnectionId, room);
        await Clients.Group(room).SendAsync("UserJoined", Context.User?.Identity?.Name);
    }

    // Покинуть комнату
    public async Task LeaveRoom(string room)
    {
        await Groups.RemoveFromGroupAsync(Context.ConnectionId, room);
        await Clients.Group(room).SendAsync("UserLeft", Context.User?.Identity?.Name);
    }

    // Вызывается автоматически при подключении клиента
    public override async Task OnConnectedAsync()
    {
        _logger.LogInformation("Client connected: {Id}", Context.ConnectionId);
        await base.OnConnectedAsync();
    }

    // Вызывается при отключении (в т.ч. при разрыве соединения)
    public override async Task OnDisconnectedAsync(Exception? exception)
    {
        _logger.LogInformation("Client disconnected: {Id}", Context.ConnectionId);
        await base.OnDisconnectedAsync(exception);
    }

Фоновые задачи и планировщик заданий — надёжно, с персистентностью

// Что такое Hangfire?

Hangfire — это библиотека для выполнения фоновых задач в .NET. Она позволяет запускать долгие операции (отправка email, генерация PDF, обработка изображений) за пределами HTTP-запроса, планировать периодические задачи (как cron), откладывать выполнение и автоматически повторять задачи при сбоях.

Главное отличие от обычных Task.Run: Hangfire сохраняет задачи в базу данных. Если сервер перезапустился — задачи не потеряются и выполнятся после старта.

Ключевое преимущество Обычный Task.Run живёт только в памяти процесса. Если сервер упал в середине отправки 10 000 писем — все они потеряны. Hangfire запишет каждую задачу в БД, и после рестарта продолжит с того места, где остановился.

// Четыре типа задач

🔥 Fire-and-Forget

Выполнить один раз как можно скорее, в фоне. Пример: отправить welcome email после регистрации.

⏳ Delayed

Выполнить через заданное время. Пример: напомнить пользователю через 3 дня о незавершённой покупке.

🔁 Recurring

Выполнять по расписанию (cron). Пример: генерировать отчёт каждый понедельник в 08:00.

🔗 Continuations

Выполнить задачу B после завершения задачи A. Пример: после конвертации видео — уведомить пользователя.

CQRS

Command Query Responsibility Segregation — разделение команд и запросов

// Что такое CQRS?

CQRS (Command Query Responsibility Segregation) — это архитектурный паттерн, суть которого в одной простой идее: операции чтения данных и операции изменения данных должны быть полностью разделены — разные модели, разные классы, разные потоки выполнения.

В традиционном подходе один и тот же сервис (или репозиторий) и читает, и записывает. CQRS говорит: Command — это всё, что изменяет состояние системы (и ничего не возвращает, кроме подтверждения). Query — это всё, что читает данные (и никогда ничего не меняет).

Принцип CQS Бертрана Мейера CQRS вырастает из более раннего принципа CQS (Command-Query Separation): «Метод должен либо что-то делать (команда), либо что-то возвращать (запрос), но не то и другое одновременно». CQRS поднимает этот принцип на уровень всей архитектуры приложения.

// CRUD vs CQRS — в чём разница?

Традиционный CRUD

Один IProductService с методами GetById, Create, Update, Delete. Одна модель Product используется везде — для чтения, записи, валидации. Просто, но плохо масштабируется при сложной логике.

CQRS

Отдельные классы CreateProductCommand, UpdateProductCommand — для изменений. Отдельные GetProductByIdQuery, GetProductsListQuery — для чтения. Каждый обработчик делает ровно одну вещь.

// Зачем это нужно?

  • Читать и писать — разные задачи. Запись требует валидации, бизнес-правил, транзакций. Чтение требует быстрых денормализованных проекций. Одна модель плохо справляется с обоими.
  • Независимое масштабирование. В большинстве систем чтений в 10–100 раз больше, чем записей. Можно масштабировать Read и Write стороны независимо.
  • Оптимизированные модели чтения. Для Query не нужен тяжёлый доменный объект — можно возвращать плоский DTO прямо из SQL без лишних джойнов через ORM.
  • Чистота и тестируемость. Каждый Handler — один класс, одна ответственность. Легко тестировать изолированно.

// Уровни применения CQRS

Уровень 1 — Логическое разделение

Одна БД, но команды и запросы — разные классы/обработчики. Самый распространённый вариант. Именно так работает связка CQRS + MediatR.

Уровень 2 — Физическое разделение

Отдельная БД для записи (нормализованная) и отдельная для чтения (денормализованная, оптимизированная). Синхронизация через события. Используется в высоконагруженных системах.

// Реализация CQRS с MediatR (Уровень 1)

Именно это сочетание используют в большинстве .NET-проектов. MediatR выступает шиной сообщений, а CQRS — архитектурным правилом разделения.

{

        var product = new Product

            Description = cmd.Description,
            Price       = cmd.Price,
            CategoryId  = cmd.CategoryId,
            Stock       = cmd.Stock,
            Slug        = slug,
            CreatedAt   = DateTime.UtcNow,
        };

        _db.Products.Add(product);
        await _db.SaveChangesAsync(ct);

        _logger.LogInformation("Product {Id} created", product.Id);
        return new CreateProductResult(product.Id, product.Name, product.Slug);
    }

public class GetProductsListQueryHandler
    : IRequestHandler<GetProductsListQuery, PagedResult<ProductSummaryDto>>
{
    private readonly AppDbContext _db;

    public async Task<PagedResult<ProductSummaryDto>> Handle(
        GetProductsListQuery query,
        CancellationToken ct)
    {
        var q = _db.Products.AsNoTracking()
            .Where(p => !p.IsDeleted);

        // Динамическая фильтрация
        if (!string.IsNullOrEmpty(query.Search))
            q = q.Where(p => p.Name.Contains(query.Search) ||
                               p.Description.Contains(query.Search));

        if (query.CategoryId.HasValue)
            q = q.Where(p => p.CategoryId == query.CategoryId);

        if (query.MinPrice.HasValue)
            q = q.Where(p => p.Price >= query.MinPrice);

        if (query.MaxPrice.HasValue)
            q = q.Where(p => p.Price <= query.MaxPrice);

        // Динамическая сортировка
        q = query.SortBy switch
        {
            "price_asc"  => q.OrderBy(p => p.Price),
            "price_desc" => q.OrderByDescending(p => p.Price),
            "rating"     => q.OrderByDescending(p => p.Reviews.Average(r => r.Rating)),
            _             => q.OrderBy(p => p.Name),
        };

        var total = await q.CountAsync(ct);

        var items = await q
            .Skip((query.Page - 1) * query.PageSize)
            .Take(query.PageSize)

public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public ValidationBehavior(IEnumerable<IValidator<TRequest>> validators)
        => _validators = validators;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken ct)
    {
        if (!_validators.Any()) return await next();

        var context = new ValidationContext<TRequest>(request);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(e => e != null)
            .ToList();

        if (failures.Any())
            throw new ValidationException(failures);

        return await next();
    }
}

        }
        catch (Exception ex)
        {
            ctx.Response.StatusCode = 500;
            await ctx.Response.WriteAsJsonAsync(new { error = "Internal server error" });
        }
    }

ORM — Object-Relational Mapping

Мост между объектным миром C# и реляционным миром SQL

// Что такое ORM?

ORM (Object-Relational Mapping, «объектно-реляционное отображение») — это технология, которая позволяет работать с реляционной базой данных, используя объекты вашего языка программирования, а не сырой SQL.

Проблема в том, что C# — объектно-ориентированный язык (данные живут в классах, объектах, коллекциях), а реляционные базы данных (SQL Server, PostgreSQL, MySQL) хранят данные в таблицах со строками и столбцами. Это фундаментальное несоответствие называется «объектно-реляционное несоответствие» (Object-Relational Impedance Mismatch). ORM решает эту проблему.

Аналогия Представьте, что вы говорите по-русски, а база данных — по-китайски. ORM — это переводчик, который переводит ваши C#-объекты в SQL-запросы туда, и результаты SQL-запросов обратно в C#-объекты — сюда.

// Без ORM vs С ORM

// ❌ БЕЗ ORM — ручной SQL и ручное маппирование

using var command = new SqlCommand(
    "SELECT Id, Name, Email, CreatedAt FROM Users WHERE Id = @Id", connection);
command.Parameters.AddWithValue("@Id", userId);

using var reader = await command.ExecuteReaderAsync();
if (await reader.ReadAsync())
{
    return new User
    {
        Id        = reader.GetInt32(0),
        Name      = reader.GetString(1),
        Email     = reader.GetString(2),
        CreatedAt = reader.GetDateTime(3),
    };
}

// ✅ С ORM (EF Core) — лаконично, типобезопасно, без SQL вручную
return await _db.Users.FindAsync(userId);

// Что ORM делает за вас

  • Маппинг классов на таблицы — класс User → таблица Users, свойство Name → столбец Name
  • Генерация SQL — LINQ-запрос переводится в оптимальный SQL SELECT/INSERT/UPDATE/DELETE
  • Управление соединениями — открытие, переиспользование, закрытие соединений с БД
  • Отслеживание изменений — ORM знает, какие объекты изменились, и генерирует только нужные UPDATE
  • Транзакции — SaveChanges() оборачивает все изменения в одну транзакцию
  • Миграции — эволюция схемы БД вместе с кодом, без ручного написания ALTER TABLE
  • Защита от SQL-инъекций — параметры автоматически экранируются

// Три подхода к работе с БД в .NET

🔷 EF Core (полный ORM)

Максимальная автоматизация. Генерирует SQL, трекинг изменений, миграции. Лучший выбор для CRUD, команд (Write-сторона CQRS), операций с доменными объектами.

⚡ Dapper (Micro-ORM)

Вы пишете SQL вручную, Dapper только маппит результат на объект. Максимальная скорость и контроль. Идеален для сложных Read-запросов (Query-сторона CQRS).

🔧 ADO.NET (без ORM)

Голый SQL, ручной маппинг. Максимальная производительность, но много boilerplate-кода. Используется только в крайних случаях или внутри Dapper.

🔀 EF Core + Dapper

Лучший из миров. EF Core — для Write-стороны и простых запросов. Dapper — для сложных аналитических Read-запросов. Именно этот подход рекомендуется для CQRS.

Entity Framework Core

Официальный ORM Microsoft — Code First, LINQ, Migrations

// Что такое EF Core?

Entity Framework Core — это официальный ORM от Microsoft для .NET. Он позволяет описывать схему базы данных в виде C#-классов (сущностей), строить запросы через LINQ, а изменения в схеме применять через систему миграций — без написания SQL вручную.

EF Core занимает центральное место в Write-стороне CQRS: Command Handler получает команду, создаёт или изменяет доменный объект, EF Core отслеживает изменения и при вызове SaveChangesAsync() генерирует и выполняет SQL-транзакцию.

// DbContext — центр управления

public class AppDbContext : DbContext
{
    public AppDbContext(DbContextOptions<AppDbContext> options) : base(options) { }

    // DbSet — представляет таблицу в БД
    public DbSet<User> Users get; set; } = null!;
    public DbSet<Order> Orders { get; set; } = null!;
    public DbSet<Product> Products { get; set; } = null!;
    public DbSet<Category> Categories { get; set; } = null!;

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Fluent API — детальная настройка маппинга (лучше, чем Data Annotations)
        modelBuilder.Entity<User>(entity =>
        {
            entity.ToTable("Users");
            entity.HasKey(u => u.Id);

            entity.Property(u => u.Name)
                  .IsRequired()
                  .HasMaxLength(200);

            entity.Property(u => u.Email)
                  .IsRequired()
                  .HasMaxLength(256);

            // Уникальный индекс на Email
            entity.HasIndex(u => u.Email).IsUnique();

            // Связь один-ко-многим (User → Orders)
            entity.HasMany(u => u.Orders)
                  .WithOne(o => o.User)
                  .HasForeignKey(o => o.UserId)
                  .OnDelete(DeleteBehavior.Cascade);
        });

        modelBuilder.Entity<Product>(entity =>
        {
            entity.Property(p => p.Price)
                  .HasColumnType("decimal(18,2)");

            entity.HasIndex(p => p.Slug).IsUnique();
        });

        // Глобальный фильтр мягкого удаления — автоматически добавляет WHERE IsDeleted = 0
        modelBuilder.Entity<User>().HasQueryFilter(u => !u.IsDeleted);
        modelBuilder.Entity<Product>().HasQueryFilter(p => !p.IsDeleted);
        modelBuilder.Entity<Order>().HasQueryFilter(o => !o.IsDeleted);

        // Применяем конфигурации из отдельных файлов
        modelBuilder.ApplyConfigurationsFromAssembly(typeof(AppDbContext).Assembly);
    }

    // Автоматическое обновление UpdatedAt при SaveChanges
    public override async Task<int> SaveChangesAsync(CancellationToken ct = default)
    {
        foreach (var entry in ChangeTracker.Entries<BaseEntity>())
        {
            if (entry.State == EntityState.Modified)
                entry.Entity.UpdatedAt = DateTimeOffset.UtcNow;
        }
        return await base.SaveChangesAsync(ct);
    }
}

// Частые ошибки EF Core

❌ N+1 Problem

Загружаете список заказов, потом в цикле — каждого пользователя. Решение: всегда используйте Include() или проекцию Select().

❌ Трекинг при чтении

Для read-only запросов (Query-сторона) всегда добавляйте AsNoTracking() — экономит память и ускоряет запрос.

❌ Lazy Loading

Не включайте Lazy Loading в продакшне — скрывает N+1 проблему. Всегда явно указывайте Include().

✓ Проверяйте SQL

Логируйте генерируемый SQL в Development: options.LogTo(Console.WriteLine, LogLevel.Information). Убедитесь, что EF строит разумный запрос.

Dapper

Micro-ORM — максимальная скорость, полный контроль над SQL

// Что такое Dapper?

Dapper — это минималистичный Micro-ORM для .NET, созданный командой Stack Overflow. Он добавляет к стандартному ADO.NET (IDbConnection) один ключевой суперпавер: автоматическое маппирование результата SQL-запроса в C#-объекты. На этом всё — никакого отслеживания изменений, никаких миграций, никакой генерации SQL.

Dapper — идеальное оружие для Query-стороны CQRS: когда нужен сложный аналитический запрос с множеством JOIN-ов, группировок и подзапросов, который EF Core либо не умеет построить оптимально, либо строит плохой SQL.

Производительность Dapper работает практически с той же скоростью, что и голый ADO.NET — overhead минимален (буквально несколько микросекунд на маппинг). EF Core добавляет больший overhead из-за трекинга изменений, кэша запросов и построения LINQ-дерева.

public class UserReadRepository
{
    private readonly string _connectionString;

    public UserReadRepository(IConfiguration config)
        => _connectionString = config.GetConnectionString("DefaultConnection")!;

    // Всегда создаём новое соединение в using — Dapper не управляет ими
    private IDbConnection CreateConnection()
        => new SqlConnection(_connectionString);

    // QueryAsync — список объектов
    public async Task<IEnumerable<UserDto>> GetAllAsync()
    {
        using var conn = CreateConnection();
        return await conn.QueryAsync<UserDto>("""
            SELECT Id, Name, Email, Role, CreatedAt
            FROM Users
            WHERE IsDeleted = 0
            ORDER BY CreatedAt DESC
            """);
    }

    // QueryFirstOrDefaultAsync — один объект
    public async Task<UserDto?> GetByIdAsync(int id)
    {
        using var conn = CreateConnection();
        return await conn.QueryFirstOrDefaultAsync<UserDto>("""
            SELECT Id, Name, Email, Role, CreatedAt
            FROM Users
            WHERE Id = @Id AND IsDeleted = 0
            """,
            new { Id = id }); // параметры — анонимный объект, защита от инъекций
    }

    // ExecuteScalarAsync — одно значение
    public async Task<int> GetTotalCountAsync()
    {
        using var conn = CreateConnection();
        return await conn.ExecuteScalarAsync<int>(
            "SELECT COUNT(*) FROM Users WHERE IsDeleted = 0");
    }

    // ExecuteAsync — INSERT/UPDATE/DELETE
    public async Task UpdateLastLoginAsync(int userId)
    {
        using var conn = CreateConnection();
        await conn.ExecuteAsync("""
            UPDATE Users
            SET LastLoginAt = @Now
            WHERE Id = @UserId
            """,
            new { UserId = userId, Now = DateTime.UtcNow });
    }
}

// Сложный запрос — QueryHandler на Dapper (Query-сторона CQRS)

public class GetOrdersDashboardQueryHandler
    : IRequestHandler<GetOrdersDashboardQuery, OrdersDashboardDto>
{
    private readonly string _connectionString;

    public GetOrdersDashboardQueryHandler(IConfiguration config)
        => _connectionString = config.GetConnectionString("DefaultConnection")!;

    public async Task<OrdersDashboardDto> Handle(
        GetOrdersDashboardQuery query,
        CancellationToken ct)
    {
        using var conn = new SqlConnection(_connectionString);

        // Сложный аналитический SQL — EF Core сгенерировал бы ужасный запрос
        var sql = """
            SELECT
                o.Status,
                COUNT(*)          AS OrderCount,
                SUM(o.TotalAmount) AS Revenue,
                AVG(o.TotalAmount) AS AvgOrderValue,
                MIN(o.CreatedAt)   AS FirstOrder,
                MAX(o.CreatedAt)   AS LastOrder
            FROM Orders o
            INNER JOIN Users u ON u.Id = o.UserId
            WHERE
                o.CreatedAt BETWEEN @DateFrom AND @DateTo
                AND o.IsDeleted = 0
                AND (@CategoryId IS NULL OR EXISTS (
                    SELECT 1 FROM OrderItems oi
                    INNER JOIN Products p ON p.Id = oi.ProductId
                    WHERE oi.OrderId = o.Id AND p.CategoryId = @CategoryId
                ))
            GROUP BY o.Status
            ORDER BY Revenue DESC
            """;

        var rows = await conn.QueryAsync<OrderStatRow>(sql, new
        {
            query.DateFrom,
            query.DateTo,
            CategoryId = (int?)query.CategoryId
        });

        return new OrdersDashboardDto
        {
            Stats      = rows.ToList(),
            TotalRevenue = rows.Sum(r => r.Revenue),
            TotalOrders  = rows.Sum(r => r.OrderCount),
        };
    }
}

// Получаем данные и метаданные пагинации за один roundtrip к БД
public async Task<PagedResult<ProductSummaryDto>> GetProductsPagedAsync(
    int page, int pageSize, string? search)
{
    using var conn = CreateConnection();

    var sql = """
        SELECT COUNT(*) FROM Products
        WHERE IsDeleted = 0
          AND (@Search IS NULL OR Name LIKE '%' + @Search + '%');

        SELECT p.Id, p.Name, p.Price, c.Name AS CategoryName, p.Stock
        FROM Products p
        INNER JOIN Categories c ON c.Id = p.CategoryId
        WHERE p.IsDeleted = 0
          AND (@Search IS NULL OR p.Name LIKE '%' + @Search + '%')
        ORDER BY p.Name
        OFFSET @Offset ROWS FETCH NEXT @PageSize ROWS ONLY;
        """;

    using var multi = await conn.QueryMultipleAsync(sql, new
    {
        Search   = search,
        Offset   = (page - 1) * pageSize,
        PageSize = pageSize
    });

    var total = await multi.ReadFirstAsync<int>();
    var items = (await multi.ReadAsync<ProductSummaryDto>()).ToList();

    return new PagedResult<ProductSummaryDto>(items, total, page, pageSize);
}

// EF Core vs Dapper — итоговое сравнение

Золотое правило в CQRS-проекте Command Handler → EF Core (трекинг, валидация, доменные события, каскады)
Query Handler → Dapper (скорость, сложный SQL, проекция в DTO без загрузки доменных объектов)
Обе библиотеки прекрасно уживаются в одном проекте и работают с одной и той же строкой подключения.

EF Core лучше для:

Создание/обновление/удаление сущностей. Управление миграциями. Каскадные операции. Отслеживание изменений. Работа с графами объектов (Order + OrderItems одновременно).

Dapper лучше для:

Аналитические запросы с 3+ таблицами. Отчёты и дашборды. Пагинация с подсчётом. Хранимые процедуры. Bulk-операции. Любой SQL, где EF Core «генерирует плохой код».
